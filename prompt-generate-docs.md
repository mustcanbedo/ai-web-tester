# AI 测试文档生成提示词

> 将以下提示词发给任意 AI（GPT/Claude/Gemini/Cursor/Windsurf 等），让它直接读取项目代码，自动生成两份测试文档供 AI Web Tester 使用。
>
> **AI Web Tester 架构**：Agent 通过 Playwright Accessibility Tree（可交互元素 ref 列表）+ 页面文本观测页面状态，由 LLM（`llm_engine.py`）决策下一步操作，`action_executor.py` 通过 `playwright_bridge.py` 执行。测试输入就是这两份文档。

---

## 使用方法

1. 在 AI IDE（Cursor/Windsurf）中打开你的**前端项目**（不是 AI Web Tester 项目）
2. 复制下面的提示词发给 AI
3. AI 会自动分析代码，生成 `core-flow.md`（功能预期文档）和 `api-docs.md`（接口文档）
4. 将生成的文件放到 AI Web Tester 的 `specs/` 目录

> 如果用的是网页版 AI（ChatGPT/Claude），把关键源码文件粘贴在提示词末尾即可。

---

## 提示词

```
你是一个专业的 QA 测试架构师。请你通读我当前项目的源代码，自动生成两份文档，用于 AI 自动化测试系统。

重要背景：这两份文档将被一个 AI 测试 Agent 消费。Agent 通过 Playwright 操作浏览器，它能看到页面的 Accessibility Tree（可交互元素列表，包含 role、name、checked 等属性）和页面文本，但看不到截图。因此文档中的每一步描述都必须精确到 Agent 能仅凭元素文本/角色定位并操作。

## 第一步：分析代码

请先阅读项目代码，重点关注以下内容：

### 路由与页面
- 路由配置文件（如 router/index.ts、App.tsx 中的 Route 定义、pages/ 目录结构）
- 每个路由对应的页面组件，了解页面功能
- 路由守卫逻辑（未登录跳转、权限拦截、重定向规则）

### API 调用
- API 请求封装（如 api/、services/、request.ts）
- 每个接口的请求方式（GET/POST）、路径、参数、返回数据结构
- 请求拦截器中的认证方式（Token Header 名称、存储位置）
- 响应拦截器中的错误处理（如 status 999 自动登出）

### 页面交互
- 表单组件：有哪些输入字段、校验规则（必填、格式、长度限制）、校验时机（失焦/提交时）、错误提示文案
- 列表组件：搜索（防抖时间）、筛选、排序、分页（每页条数）、轮询刷新
- 弹窗/对话框：触发条件、标题文本、确认/取消按钮文本、destructive 样式
- 下拉选择器：选项列表来源、默认选中项、是否支持搜索
- 复选框/开关：默认状态、是否必选、关联的提交校验
- 特殊交互：文件上传、拖拽、Tab 切换、步骤向导、倒计时

### 认证与权限
- 登录方式（账号密码/手机验证码/SSO 等）
- 登录表单的完整字段（国家区号选择器、手机号、验证码、同意条款 checkbox 等）
- Token 存储位置（localStorage/sessionStorage/cookie）和字段名
- 路由守卫的具体行为（重定向目标、是否弹登录弹窗、redirect 信息存储）

### 数据模型
- TypeScript 类型定义（interfaces/types）
- 状态枚举及其对应的 UI 文案（如 status=4 显示"成功"）
- 实体之间的层级关系（如项目包含多个任务）

### UI 文案提取
- i18n 国际化文件（中文翻译）
- 组件中硬编码的文本（按钮文字、placeholder、toast 提示、错误信息）
- 页面标题（document.title / meta.title）

## 第二步：生成两份文档

### 文档一：core-flow.md（功能预期文档）

这份文档描述 AI 测试 Agent 需要执行的完整测试流程。**Agent 只能看到 Accessibility Tree 和页面文本**，所以每一步都必须用元素的可见文本、角色或 placeholder 来描述目标元素。

**格式要求**：

1. 文档顶部用引用块说明用途和登录凭据占位符
2. 用 Markdown 格式，按业务流程分章节（`## 流程一：xxx`、`## 流程二：xxx`）
3. 每个流程开头写明 **前置条件**（如"用户已登录"、"已清除 localStorage"）
4. 每个流程下用**编号列表**描述具体操作步骤
5. 每步操作必须包含：
   - **具体动作**：点击什么按钮（写出按钮的精确文本）、在哪个输入框填写什么（写出 placeholder）
   - **预期结果**：操作后页面应该出现什么变化（包括 UI 变化、状态变化、存储变化）
6. 对于**下拉菜单、弹窗、Tab 切换**等复杂交互，必须拆分为多个子步骤（先 click 展开 → 等待列表出现 → 再 click 目标选项）
7. 每个流程末尾列出 `### 异常场景`，覆盖各种非法输入和错误情况

**内容深度要求（这是生成高质量文档的关键）**：

**A. 精确的 UI 文案**：
- 所有按钮文字、placeholder、toast 提示、错误信息都必须从代码/i18n 中提取原文，用 **粗体** 标注
- 格式化文本用模板标注，如 **"{seconds}秒重新发送"**
- 输入框要标注 `placeholder`、`maxlength`、`inputmode` 等关键属性

**B. 状态变化描述**：
- 按钮的 disabled/loading 状态变化（如"按钮变为 disabled，文字变为倒计时"）
- 加载状态（loading 动画、加载文字如 **"加载数据集..."**）
- 提交状态（"按钮文字变为 **'登录中...'**"）

**C. 存储层验证**：
- `localStorage` 中的 token/user 字段变化
- `sessionStorage` 中的重定向信息
- Cookie 的设置和清除

**D. URL 和路由变化**：
- 每次导航后的预期 URL 路径和 query 参数
- SPA 路由切换时 URL 是否变化
- 弹窗操作时 URL 是否保持不变

**E. 响应式布局差异**（如果有）：
- 不同屏幕断点下的布局变化（如移动端的悬浮按钮 vs PC 端的工具栏按钮）

**F. 数据一致性验证**（独立流程）：
- 列表 → 详情的数据一致性
- CRUD 操作后的数据刷新验证
- 跨页面的数据同步（如重命名后在另一个页面也能看到新名称）
- 数量统计验证（如项目的任务数 = 实际任务列表条数）

**G. 认证守卫验证**（独立流程）：
- 未登录访问受保护页面的行为
- Token 失效后的自动登出行为
- 登录后的自动重定向

**H. 异常场景必须穷举**：
- 每个输入字段的校验规则和对应的错误提示文案
- 空值提交
- 格式不合法
- 边界值（最小/最大长度）
- 网络错误
- 权限不足

**流程编排顺序**：
1. 登录流程（必须是第一个，包含完整的表单交互）
2. 核心 CRUD 流程（按业务主线排列）
3. 数据一致性验证（独立流程，交叉验证各页面数据）
4. 认证守卫验证（独立流程）
5. 基础信息验证（页面标题、favicon 等）

**示例格式**：

```markdown
# XXX 系统核心功能预期文档

> 本文档供 AI 自动化测试系统使用。所有按钮文本、输入框 placeholder 均提取自源码 i18n 中文文案。
> 登录凭据占位符：手机号 `13800138000`，验证码由短信获取（4 位数字）。

---

## 流程一：用户登录（手机验证码登录）

**前置条件**：浏览器已清除 `localStorage` 中的 `token` 和 `user`。

1. 打开首页 `/`，页面正常加载。
2. 点击导航栏右侧的 **"登录/注册"** 按钮。
3. 预期结果：弹出登录模态对话框，URL 路径**不变**（仍为 `/`），弹窗标题显示 **"欢迎来到 XXX"**。
4. **选择国家区号**：在手机号输入框左侧，点击国旗/区号下拉按钮，展开国家列表。
5. 在国家列表中找到并点击 **China**（区号 `+86`）。
6. 预期结果：下拉收起，输入框左侧显示中国国旗和 `+86`。
7. **输入手机号**：在手机号输入框（placeholder 为 **"请输入手机号"**）中输入纯数字手机号 `13800138000`（不带 `+86` 前缀）。
8. 预期结果：手机号格式校验通过（中国手机号正则 `/^1[3-9]\d{9}$/`），无红色错误提示。
9. 点击 **"获取验证码"** 按钮。
10. 预期结果：
    - 按钮文字变为 **"{seconds}秒重新发送"**（从 60 开始倒数），按钮变为 disabled。
    - 页面出现 toast 提示 **"验证码已发送"**。
11. 在验证码输入框（placeholder 为 **"验证码"**，`maxlength="4"`）中输入 4 位验证码。
12. 勾选 **"我已同意条款和政策"** 复选框（如果尚未勾选）。
13. 点击 **"登录/注册"** 按钮。
14. 预期结果：
    - 按钮显示加载状态，文字变为 **"登录中..."**。
    - 登录成功后 toast 提示 **"登录/注册成功"**，弹窗自动关闭。
    - `localStorage` 中新增 `token`（JWT）和 `user`（JSON）。
    - 导航栏 **"登录/注册"** 按钮消失，显示用户头像或昵称。

### 异常场景

- **手机号为空**：点击"获取验证码"时，显示红色提示 **"请输入手机号"**。
- **手机号格式不合法**：输入非 11 位数字，显示 **"请输入有效的手机号码"**。
- **验证码为空**：点击"登录/注册"时，显示 **"请输入验证码"**。
- **验证码格式错误**：输入非 4 位数字，显示 **"请输入4位验证码"**。
- **未勾选协议**：点击"登录/注册"时，显示 **"请先同意条款和协议"**。
```

### 文档二：api-docs.md（接口文档）

这份文档描述系统的后端 API 接口，AI 测试 Agent 会用它来做**三层数据验证**（前端展示 vs API 返回 vs 数据逻辑）。

**格式要求**：

1. 用 Markdown 格式，按功能模块分章节
2. 每个接口包含：
   - **请求方式和路径**：如 `POST /api/v1/order/list`
   - **Content-Type**：如 `application/json` 或 `multipart/form-data`
   - **描述**：一句话说明接口用途
   - **请求参数**：用表格列出（字段名、类型、是否必填、说明）
   - **关键返回数据**：用 JSON 示例展示，标注重要字段含义
3. 如果有认证机制，说明 Token 传递方式（Header 名称）

**必须包含的最后一节：数据逻辑关联**

这一节非常重要，描述接口之间的数据关系，供 AI Agent 做交叉验证：

- **层级关联**：如"项目列表中的 `id` 对应任务列表的 `project_id`"
- **数量校验**：如"项目的 `task_count` 应等于该项目下任务列表的 `total`"
- **名称一致性**：如"任务详情中的 `project_name` 应与项目列表中对应项目的名称一致"
- **列表与详情一致性**：如"列表接口和详情接口返回的同一条数据，所有字段应一致"
- **状态与数据关联**：如"订单状态为'已发货'时，`tracking_number` 不应为空"
- **CRUD 幂等性**：如"修改名称后重新查询，名称应为新值"；"删除后重新查询，列表不应包含已删除项"
- **权限校验**：如"普通用户调用管理员接口应返回 403"

**示例格式**：

```markdown
### 1.1 获取订单列表

- **请求**: `POST /api/v1/order/list`
- **Content-Type**: `application/json`
- **描述**: 分页获取订单列表，支持状态筛选和搜索。
- **请求参数**:

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| page | number | 是 | 页码 |
| size | number | 是 | 每页条数 |
| status | number | 否 | 订单状态：1-待付款, 2-已付款, 3-已发货, 4-已完成 |
| keyword | string | 否 | 订单号/商品名搜索 |

- **关键返回数据**:
```json
{
  "code": 0,
  "data": {
    "list": [
      {
        "id": 1,
        "order_no": "ORD20260101001",
        "status": 2,
        "total_amount": 299.00,
        "created_at": "2026-01-01 10:00:00"
      }
    ],
    "total": 100,
    "page": 1,
    "size": 20
  }
}
```
```

## 第三步：输出

1. 直接生成两个文件：`core-flow.md` 和 `api-docs.md`
2. 不要输出解释说明，直接输出文档内容
3. 操作步骤中的按钮文本、输入框 placeholder、toast 提示、错误信息等，**必须**从代码/i18n 中提取真实值，用粗体标注，不要编造
4. 登录凭据用占位符表示（如手机号用 `13800138000`、密码用 `TestPass123`）
5. 如果某个交互细节无法从代码确认，用 `<!-- TODO: 请确认 -->` 标注
6. 每个异常场景都要写出精确的错误提示文案

## 补充信息（可选）

如果你觉得有必要，可以补充以下信息帮助 AI 更准确地生成：
- 项目线上地址（用于填写测试目标 URL）
- 重点关注的模块或已知 bug 区域
- 测试账号凭据
```

## 使用场景

### 场景 1：AI IDE 中直接使用（推荐）

在 Cursor / Windsurf 中打开前端项目，直接发送上面的提示词，AI 会自动读取代码并生成两份文档。

### 场景 2：网页版 AI

把关键源码文件粘贴在提示词末尾：

```
【粘贴上面的提示词】

以下是我的项目关键文件：

--- router/index.ts ---
【粘贴路由配置】

--- api/index.ts ---
【粘贴 API 封装】

--- i18n/zh-CN.json ---
【粘贴中文国际化文件】

--- pages/Login.tsx ---
【粘贴登录页组件】

...其他关键页面组件...
```

### 场景 3：有 Swagger/OpenAPI 文档

如果项目已有接口文档，可以追加：

```
【粘贴上面的提示词】

补充：以下是项目的 Swagger JSON，请优先基于此生成 api-docs.md：
【粘贴 swagger.json 内容】
```

生成的文件保存到 AI Web Tester 的 `specs/` 目录即可使用。
