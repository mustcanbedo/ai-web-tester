# AI Web Tester — Bad Cases & 技术难点

> 基于 19 份测试报告（`reports/` 目录）的系统性分析
> 更新时间：2026-03-19

本文档记录的是 **测试工具本身的问题**，包括：
- **误报（False Positive）**：Agent 报告为产品 Bug，但实际是工具自身能力不足导致的误判
- **技术难点**：当前架构下尚未解决或难以解决的问题

---

## 一、误报清单（False Positives）

### FP-1 ⭐ Popover 菜单"未弹出" → 实为工具点击失效

| 项目 | 内容 |
|------|------|
| **报告中的描述** | "点击 `···` 按钮后 Popover 菜单未弹出"、"下载按钮点击后下载菜单未弹出" |
| **出现频率** | 19 份报告中约 **12 份** 报告了此问题，是最高频误报 |
| **误报原因** | **根因已更正（2026-03-19 验证）**：Radix Vue `PopoverTrigger` 实际使用标准 `@click` 事件（非 `@pointerdown`），Playwright `locator.click()` 在登录态下完全能正常打开 Popover。真正的阻断原因是：Agent cookies 失效后，页面出现 `z-9999` 登录弹窗遮罩层（`div.fixed.inset-0`），拦截了所有 pointer events，导致 `locator.click()` 超时失败。Agent 将此误判为"Popover 组件点击失效" |
| **真实情况** | 被测应用 Popover 功能正常。登录后 `locator.click()` 和 `el.click()` 均可正常弹出菜单 |
| **涉及的误报 Bug** | • "更多"按钮(···) Popover 菜单未弹出（被报为 P0/P1）<br>• 下载按钮点击后下载菜单未弹出（P1）<br>• 任务卡片交互按钮全部失效（P1）<br>• 3DGS 任务缺少下载选项（P1）—— 实际是菜单没打开所以看不到 |
| **解决状态** | � 已修复：① click 方法增加 `el.click()` 降级（绕过遮罩层 hit-testing）② 对 `aria-haspopup` 元素验证 `data-state` 并重试 ③ system prompt 增加"遮罩层优先处理"硬性规则 |

### FP-2 ⭐ 项目卡片"不可点击/不可访问" → 实为 AX Tree 未暴露交互元素

| 项目 | 内容 |
|------|------|
| **报告中的描述** | "项目卡片在 Accessibility Tree 中无任何可交互元素"、"项目卡片完全不可访问"、"无法从项目列表进入训练任务页面" |
| **出现频率** | 约 **4 份** 报告将此报为 P0 阻断问题 |
| **误报原因** | 项目卡片的点击跳转通过 Vue `@click` 绑定在 `<div>` 上（非标准 `<a>` 或 `<button>`），Playwright AX Tree 不将其识别为可交互元素。Agent 在 refs 中找不到可点击的项目名称/预览图，反复尝试后判定为"产品缺陷" |
| **真实情况** | 被测应用项目卡片点击跳转功能正常 |
| **涉及的误报 Bug** | • 项目卡片缺少可访问操作入口（P0）<br>• 项目卡片完全不可访问（P0）<br>• 无法进入训练任务页面（P0）<br>• 项目卡片名称不可点击（P2） |
| **解决状态** | 🟡 部分解决（已有非标准可点击元素检测），但仍不稳定，见 TD-2 |

## 三、技术难点（待解决）

### TD-1 🔴 Radix Vue Popover / Dialog 点击触发

**难度**：⭐⭐⭐⭐⭐（最高优先级）

**问题本质**：Playwright 原生鼠标事件无法可靠触发 Radix Vue 的 Popover/Dialog 组件。

**已尝试**：
- `locator.click()` → 不生效
- JS 降级点击（pointerdown→mousedown→pointerup→mouseup→click）→ 部分场景不生效

**待尝试方向**：
1. 直接 `dispatchEvent(new PointerEvent('pointerdown', {bubbles: true, composed: true}))` 到精确的 DOM 节点
2. 通过 CDP 协议发送 `Input.dispatchMouseEvent` 模拟更底层的输入
3. 检查 Radix Vue 内部是否依赖 `event.isPrimary` 或 `event.pointerId` 等属性
4. hover → 短暂延迟 → click 的组合操作
5. 直接调用 Radix Vue 组件的内部状态切换方法（`open = true`）作为最终降级方案

**影响范围**：流程三约 60% 的测试用例因此被阻断，是当前最大的误报来源。

---

### TD-2 🟠 非标准可点击元素的识别与交互

**难度**：⭐⭐⭐⭐

**问题本质**：Vue 应用中大量使用 `<div @click>` 而非 `<a>` / `<button>`，Playwright AX Tree 不将其识别为可交互元素。

**当前方案**：
- 已实现 `cursor:pointer` + Vue `_vei` / React `__reactProps$` 检测
- 检测到后追加到 refs 列表

**未解决的边界情况**：
- 事件委托：父元素绑定 `@click`，子元素无直接事件 → 检测遗漏
- 动态绑定：组件 mount 后异步绑定事件 → 快照时尚未绑定
- CSS pointer-events 被覆盖：`cursor:pointer` 在 computed style 中但被子元素遮挡
- 项目卡片整体可点击但仅部分区域有视觉反馈 → Agent 不知道该点哪里

---

### TD-3 🟠 Agent 重复执行已完成步骤 / 陷入循环

**难度**：⭐⭐⭐

**问题本质**：Agent 在某个操作失败后，反复尝试相同或类似操作，浪费步数（每次 thinking 描述不同，文本去重无效）。

**当前方案**：
- `done_items_in_current_flow` 注入上下文
- 卡死检测（连续 6 步失败 / 2 URL 来回跳）→ 强制跳转

**未解决**：
- 6 步触发阈值太高，前 5 步浪费
- 上下文裁剪后 Agent "忘记"已做过的事
- Agent 倾向于"重试"而非"跳过"，即使文档明确允许跳过

---

### TD-4 🟠 Agent 违反硬性约束（进入 projectId=25）

**难度**：⭐⭐

**问题本质**：`core-flow.md` 中的禁止规则在上下文裁剪后被丢失，或 Agent 按"选择第一个项目"逻辑自动选中了禁止项目。

**解决方向**：
- 在 system prompt（不会被裁剪）中硬编码禁令
- 在 `action_executor.py` 的 `navigate` 中加 URL 黑名单拦截
- 在 `core-flow.md` 中将禁令提升到文档最顶部

---

### TD-5 🟡 LLM 上下文窗口管理

**难度**：⭐⭐⭐

**问题本质**：测试 100+ 步后消息历史被大幅裁剪，早期规则/进度信息丢失。

**影响**：
- Agent "忘记"已完成的子步骤 → 重复执行（关联 TD-3）
- Agent "忘记"禁止规则 → 违规操作（关联 TD-4）
- 流程进度判断不准确

**解决方向**：
- 持久记忆机制：关键决策/发现写入独立 memory，不随消息裁剪
- 已完成流程做摘要压缩而非直接删除
- 将关键约束从流程文档提升到 system prompt

---

### TD-6 🟡 `fill` 操作对 Vue `v-model` 的兼容性

**难度**：⭐⭐

**问题本质**：Playwright 的 `fill()` 方法设置 `input.value` 后触发的事件可能不被 Vue `v-model` 监听到，导致组件状态未更新。

**表现**：
- 搜索框填入关键词后列表未刷新
- 表单填入数据后提交时字段为空

**解决方向**：
- fill 后手动 dispatch `input` + `change` 事件
- 改用 `type` 逐字输入（更接近真实用户，但更慢）
- 在 `playwright_bridge.py` 中统一处理 fill 后事件触发

---

### TD-7 🟡 iframe 内容无法通过 AX Tree 感知

**难度**：⭐⭐⭐

**问题本质**：3D 查看器（点云/3DGS/Mesh/CAD）通过 iframe 加载，AX Tree 快照无法穿透 iframe 获取内部 DOM。

**表现**：
- Agent 访问查看器页面后 AX Tree 仅显示外层 4 个基础元素
- Agent 判定"页面完全空白"（可能实际 iframe 已加载但不可见于 AX Tree）

**解决方向**：
- 切换到 iframe 的 frame context 再获取 AX Tree
- 使用 `execute_js` 检查 iframe 的 `contentDocument` 状态
- 在 core-flow.md 中标注 iframe 页面的验证方式（如检查 iframe src 是否正确、HTTP 状态码等）
